# Hermes × Codex × 量化投资系统：职责分工与模拟交易路线 v0.1

## 0. 总原则

本系统的目标不是让大模型直接预测市场、自动赚钱，也不是让 LLM 直接控制实盘资金。第一阶段目标是：

1. 建立一个可复现的量化研究系统；
2. 建立一个低成本的信息收集与归档系统；
3. 建立一个有风控约束的模拟交易系统；
4. 通过 2～3 个月纸面交易验证流程稳定性；
5. 让 Codex 只承担高价值工程与审计任务，避免被新闻摘要、文档归类、低价值检索任务消耗额度。

核心定位：

* Hermes 是常驻入口、低成本信息处理层、任务路由层、归档层。
* Codex 是高价值工程代理、代码审计代理、复杂研究任务执行器。
* 普通程序是确定性执行层，负责数据抓取、回测、模拟交易、风控规则、日志落盘。
* 人是最终决策者，尤其是涉及实盘、资金、权限、API key、风控阈值、策略上线时。

一句话原则：

> Hermes 负责吞吐量，Codex 负责关键质量，交易执行必须被规则系统和人工确认约束。

## 1. 系统总体架构

推荐架构：

```text
微信 / Obsidian / 命令行
        ↓
Hermes + 本地 Qwen3.6 27B
        ↓
任务分类器 / 路由器
        ↓
低价值任务：Hermes 自己处理
高价值工程任务：调用 Codex MCP Server
        ↓
Codex：读仓库、改代码、跑测试、审计回测逻辑
        ↓
结果返回 Hermes
        ↓
归档到 Obsidian / 生成日报 / 等待人工确认
```

更具体一点：

```text
[用户入口]
  - 微信私聊
  - Obsidian 笔记
  - CLI 指令

[Hermes 层]
  - 解析自然语言任务
  - 收集新闻和数据
  - 归类文档
  - 维护策略想法库
  - 维护实验队列
  - 维护模拟交易日志
  - 判断是否升级给 Codex

[Codex MCP 层]
  - 进入指定 repo
  - 阅读代码
  - 修改代码
  - 运行测试
  - 审计回测
  - 检查未来函数
  - 检查数据泄漏
  - 生成实验报告

[量化系统层]
  - 数据库
  - 因子计算
  - 回测引擎
  - 模拟交易账户
  - 风控模块
  - 报告模块

[人工控制层]
  - 是否允许实盘
  - 是否允许下单
  - 是否允许改仓位
  - 是否允许修改风控
  - 是否允许接入真实券商 API
```

## 2. Hermes 与 Codex 的职责边界

### 2.1 Hermes 负责什么

Hermes 的价值不是"比 Codex 更聪明"，而是低成本、常驻、可堆任务、适合做缓冲层。

Hermes 负责：

1. 信息入口

   * 接收微信消息；
   * 接收 Obsidian 笔记；
   * 接收命令行任务；
   * 把模糊想法变成结构化任务。

2. 新闻和资料处理

   * 抓取新闻；
   * 粗略摘要；
   * 分类到行业、公司、宏观、政策、财报、风险事件；
   * 标记是否可能影响某个策略假设；
   * 生成每日/每周市场简报。

3. 文档归档

   * 把新闻、研报、回测报告、策略想法归档到 Obsidian；
   * 维护策略假设库；
   * 维护实验结果索引；
   * 维护失败案例库。

4. 策略想法整理

   * 把"我想试试低波动加动量"整理成实验卡片；
   * 指定标的池、时间范围、调仓频率、手续费、滑点、评价指标；
   * 准备 Codex 可执行的任务说明。

5. Codex 任务筛选

   * 判断任务是否值得消耗 Codex 额度；
   * 生成 Codex Job；
   * 选择 read-only 或 workspace-write 模式；
   * 接收 Codex 输出后归档。

6. 模拟交易日报

   * 当前持仓；
   * 今日盈亏；
   * 累计收益；
   * 最大回撤；
   * 换手率；
   * 异常交易；
   * 新闻与持仓的可能关联。

Hermes 不应该负责：

* 直接修改核心交易代码；
* 直接改风控参数；
* 直接接触券商 API key；
* 直接下实盘单；
* 在没有 Codex 审计的情况下上线新策略；
* 把新闻情绪直接转成交易指令。

---

### 2.2 Codex 负责什么

Codex 是高价值工程代理。它适合做需要强推理、强代码能力、强审计能力的任务。

Codex 负责：

1. 代码开发

   * 建立量化仓库；
   * 写数据管道；
   * 写因子计算模块；
   * 写回测模块；
   * 写模拟交易模块；
   * 写风控模块；
   * 写报告生成模块。

2. 代码审计

   * 检查未来函数；
   * 检查幸存者偏差；
   * 检查复权错误；
   * 检查财报发布时间滞后；
   * 检查调仓日和成交日错位；
   * 检查交易成本、滑点、停牌、涨跌停处理；
   * 检查测试覆盖。

3. 实验复现

   * 运行指定回测；
   * 生成可复现配置；
   * 固定随机种子；
   * 输出指标；
   * 写实验报告；
   * 对比不同策略版本。

4. 框架重构

   * 把 notebook 实验整理成 repo；
   * 把脚本整理成模块；
   * 把临时代码变成可测试代码；
   * 把数据接口封装成统一 schema。

5. 风控实现

   * 最大单股权重；
   * 最大行业暴露；
   * 最大换手率；
   * 最大回撤熔断；
   * 单日亏损限制；
   * 禁止杠杆；
   * 禁止做空；
   * 禁止期权和高风险衍生品。

Codex 不应该负责：

* 长期新闻摘要；
* 大量文档分类；
* 低价值检索；
* 每日重复性归档；
* 自动实盘下单；
* 未经人工确认修改风控；
* 直接读取或保存券商 API 密钥。

### 3. 什么时候 Hermes 应该调用 Codex

### 3.1 不调用 Codex 的情况

以下任务 Hermes 自己处理：

* 新闻摘要；
* 文档分类；
* Obsidian 归档；
* 市场日报；
* 每日持仓摘要；
* 策略想法初步整理；
* 粗略行业新闻归类；
* 简单指标解释；
* 把微信消息转成任务卡片。

判断标准：

> 不涉及代码修改、不涉及仓库审计、不涉及复杂回测、不影响真实资金，就不要调用 Codex。

---

### 3.2 调用 Codex read-only 的情况

以下任务调用 Codex，但默认只读：

```text
sandbox = read-only
approval = on-request 或 never
allow_write = false
```

适合任务：

* 审查某个策略是否可能有未来函数；
* 审查回测框架是否有数据泄漏；
* 读取仓库并生成结构说明；
* 检查某个因子定义是否和文档一致；
* 检查测试覆盖是否不足；
* 解释某次回测结果为什么异常；
* 检查模拟交易日志是否和策略信号一致。

示例任务：

```text
请进入 quant-research 仓库，只读检查 backtest 模块。
重点检查：
1. 信号生成日期是否早于成交日期；
2. 财报数据是否按披露日期滞后；
3. 调仓日、成交日、持仓日是否对齐；
4. 是否存在幸存者偏差；
5. 是否正确处理手续费和滑点。
不要修改任何文件。
输出：问题列表、风险等级、建议修法。
```

---

### 3.3 调用 Codex workspace-write 的情况

以下任务可以让 Codex 修改代码，但必须满足前置条件：

```text
sandbox = workspace-write
approval = on-request
allow_write = true
```

前置条件：

1. 当前 git 工作区必须干净；
2. 必须在独立分支或 worktree；
3. 不允许直接改 main；
4. 不允许改 API key、secrets、真实交易配置；
5. 不允许改风控上限；
6. 修改后必须跑测试；
7. 输出必须包含 diff 摘要、测试结果、风险说明。

适合任务：

* 新增一个因子；
* 新增一个回测实验；
* 修复测试失败；
* 写模拟交易账本；
* 写风控模块；
* 统一数据 schema；
* 把 notebook 改成可复现脚本；
* 生成报告模板。

示例任务：

```text
请在 quant-research 仓库中新建一个实验：
策略：月度动量 + 低波动过滤
范围：先用示例数据或已有本地数据
要求：
1. 新增 strategy 配置；
2. 新增因子计算函数；
3. 新增回测脚本；
4. 新增至少 3 个单元测试；
5. 检查信号日期和成交日期不能相同；
6. 运行相关测试。
禁止：
- 不要接真实券商 API；
- 不要写入真实账户信息；
- 不要修改全局风控上限。
```

---

### 3.4 必须人工确认的情况

以下情况 Hermes 不能自动交给 Codex，Codex 也不能自动执行：

* 实盘下单；
* 修改真实资金配置；
* 修改券商 API 权限；
* 保存或读取 API secret；
* 开启杠杆；
* 做空；
* 买期权；
* 修改最大仓位；
* 修改最大回撤熔断；
* 删除历史数据；
* 自动 push 到远程 main；
* 自动部署实盘服务。

这些任务必须人工确认，并且最好有二次确认。

## 4. Codex MCP Server 作为桥梁的设计

推荐把 Codex 作为 Hermes 的 sub-agent，而不是普通模型。

Hermes 不应该"模拟人操作 Codex 终端"，而应该通过 MCP 方式把结构化任务交给 Codex。

基本逻辑：

```text
Hermes
  ↓
生成 Codex Job
  ↓
调用 Codex MCP Server
  ↓
Codex 在指定 repo 中执行
  ↓
Codex 返回报告 / patch / 测试结果
  ↓
Hermes 归档结果
```

建议每个 Codex Job 都包含以下字段：

```yaml
job_id: 2026-06-27-quant-audit-001
repo: ~/projects/quant-research
mode: read-only
priority: high
task_type: audit
created_by: hermes
requires_human_approval: false

context:
  - 当前任务来自量化系统模拟交易项目
  - 用户希望先做 2～3 个月模拟交易
  - 禁止接入真实资金
  - 禁止自动下实盘单

task:
  - 检查 backtest 模块是否存在未来函数
  - 检查调仓日期和成交日期是否错位
  - 检查手续费和滑点是否进入收益计算

output_required:
  - 问题列表
  - 风险等级
  - 文件路径
  - 建议修改方案
  - 是否需要进一步修改代码

forbidden:
  - 不要修改文件
  - 不要访问网络
  - 不要读取 secrets
  - 不要生成实盘交易指令
```

workspace-write 任务则额外要求：

```yaml
mode: workspace-write
branch_required: true
must_run_tests: true
must_report_diff: true
must_not_touch:
  - .env
  - secrets/
  - live_trading_config.yaml
  - broker_keys.json
  - risk_limits_production.yaml
```

## 5. 两到三个月模拟交易需要搭建什么

第一阶段不要追求复杂模型。目标是验证流程，而不是立刻赚钱。

建议先做：

* 只做股票；
* 只做现货；
* 不做杠杆；
* 不做期权；
* 不做做空；
* 不做高频；
* 不做盘中自动交易；
* 先用日频数据；
* 先做纸面交易；
* 所有交易都要可解释、可回放、可审计。

### 5.1 数据层

必须有：

1. 行情数据

   * 开盘价；
   * 最高价；
   * 最低价；
   * 收盘价；
   * 成交量；
   * 成交额；
   * 复权因子。

2. 交易日历

   * 交易日；
   * 节假日；
   * 半日市；
   * 调仓日；
   * 成交日。

3. 标的池

   * 当前股票池；
   * 历史成分；
   * 退市股票；
   * 停牌股票；
   * 是否可交易。

4. 公司行为

   * 分红；
   * 拆股；
   * 合股；
   * 配股；
   * 退市；
   * 停牌。

5. 基本面数据

   * 财报；
   * 市值；
   * PE；
   * PB；
   * ROE；
   * 负债率；
   * 营收增速；
   * 净利润增速。

6. 新闻和事件数据

   * 公司新闻；
   * 行业新闻；
   * 宏观新闻；
   * 政策新闻；
   * 财报公告；
   * 重大风险事件。

关键点：

> 量化系统最先坏掉的地方通常不是模型，而是数据。

---

### 5.2 策略研究层

初期不要直接上复杂深度学习或强化学习。建议从最基础的因子开始：

1. 动量因子

   * 过去 3 个月收益；
   * 过去 6 个月收益；
   * 过去 12 个月收益。

2. 反转因子

   * 过去 5 日跌幅；
   * 过去 20 日跌幅；
   * 短期超跌反弹。

3. 低波动因子

   * 过去 20 日波动率；
   * 过去 60 日波动率；
   * 过去 120 日波动率。

4. 价值因子

   * PE；
   * PB；
   * 股息率；
   * 自由现金流收益率。

5. 质量因子

   * ROE；
   * 毛利率；
   * 负债率；
   * 盈利稳定性。

6. 流动性因子

   * 成交额；
   * 换手率；
   * 买卖价差；
   * 冲击成本。

7. 新闻/情绪因子

   * 新闻数量；
   * 负面新闻比例；
   * 财报事件；
   * 政策事件；
   * 宏观风险事件。

初期建议优先做：

```text
动量 + 低波动
价值 + 质量
短期反转
高股息 + 低波动
```

不要一开始就做：

```text
LLM 直接判断买卖
强化学习自动交易
复杂多因子神经网络
盘中高频交易
期权策略
杠杆策略
```

原因是这些东西很容易过拟合，也很难判断失败是数据问题、模型问题、交易成本问题，还是市场本来无效。

---

### 5.3 回测层

回测层必须严格。

至少要支持：

1. 策略配置

   * 标的池；
   * 时间范围；
   * 调仓频率；
   * 因子；
   * 权重方法；
   * 交易成本；
   * 滑点；
   * 最大持仓数；
   * 最大单股权重。

2. 信号生成

   * 哪一天生成信号；
   * 用到哪些数据；
   * 数据截止到哪一天；
   * 什么时候成交。

3. 交易模拟

   * 买入；
   * 卖出；
   * 调仓；
   * 手续费；
   * 滑点；
   * 停牌；
   * 无法成交；
   * 部分成交。

4. 组合净值

   * 每日净值；
   * 每日收益；
   * 累计收益；
   * 最大回撤；
   * 夏普比率；
   * 波动率；
   * 换手率。

5. 审计

   * 是否未来函数；
   * 是否幸存者偏差；
   * 是否使用了未披露财报；
   * 是否错用了复权价格；
   * 是否忽略退市股票；
   * 是否低估交易成本。

必须建立一个回测审计清单。Codex 的高价值任务之一就是反复检查这个清单。

---

### 5.4 模拟交易层

模拟交易不是简单把回测结果搬过来。模拟交易要像真实账户一样记账。

必须有：

1. 虚拟现金账户；
2. 虚拟持仓；
3. 订单表；
4. 成交表；
5. 每日市值；
6. 每日盈亏；
7. 手续费；
8. 滑点；
9. 拒单记录；
10. 策略信号与实际订单的对应关系。

推荐最小表结构：

```text
orders:
  order_id
  strategy_id
  symbol
  side
  target_weight
  target_quantity
  order_type
  signal_date
  submit_time
  status
  reason

trades:
  trade_id
  order_id
  symbol
  side
  quantity
  price
  commission
  slippage
  trade_time

positions:
  date
  symbol
  quantity
  cost_basis
  market_price
  market_value
  unrealized_pnl

portfolio:
  date
  cash
  market_value
  total_equity
  daily_return
  drawdown
```

第一阶段可以先做"内部模拟账户"，不急着接券商 paper trading。
第二阶段再接 IBKR、Alpaca 或其他券商的纸面交易接口。

---

### 5.5 风控层

风控必须先于策略收益。

第一阶段默认规则可以设为：

```text
禁止实盘
禁止杠杆
禁止做空
禁止期权
禁止盘中自动交易
最大单股权重：10%
最大行业权重：30%
最大日换手：30%
最大持仓数：10～30
单日模拟亏损超过阈值：停止调仓
组合最大回撤超过阈值：停止新增仓位
任何交易前必须能追溯到 strategy_id 和 signal_id
```

关键原则：

> 策略可以错，但风控不能缺。

---

### 5.6 报告层

每天至少生成：

1. 今日组合净值；
2. 今日收益；

Hermes 负责生成这些报告。
Codex 只在报告异常、代码异常、回测异常时介入。

### 第 3～4 周：第一个可回测策略

### 第 5～6 周：新闻与事件层

* 建立新闻分类；

交付物：

* 新闻分类规则；
* Hermes 新闻归档流程。

### 第 7～8 周：模拟交易

目标：从回测进入纸面交易。

任务：

* 每日生成信号；
* 写入模拟订单；
* 按规则模拟成交；
* 更新持仓；
* 生成日报；
* 记录拒单原因；
* 记录风控触发情况。

每日流程：

```text
盘后更新数据
  ↓
生成信号
  ↓
风控检查
  ↓
生成模拟订单
  ↓
模拟成交
  ↓

  ↓

```

这个阶段仍然不建议接真实资金。

---

### 第 9～12 周：稳定性验证

目标：判断系统是否可靠，而不是判断策略是否必然赚钱。

重点观察：

* 数据是否稳定更新；
* 策略是否每天能正常生成信号；
* 订单是否能追溯；
* 风控是否真的生效；
* 报告是否能解释持仓变化；
* 回测和模拟交易是否存在明显偏差；
* Hermes 是否过度调用 Codex；
* Codex 任务是否都能复现。

结束时输出：

```text
2～3 个月模拟交易总结报告

  - 策略表现
  - 最大回撤
  - 换手率
  - 数据异常
  - 风控触发
  - 新闻解释效果
  - Codex 调用次数
  - 失败案例
  - 是否值得进入小资金人工确认实盘
```

## 7. 可以借鉴的 GitHub 项目思路

### 7.1 TradingAgents

TradingAgents 的核心价值不是"它能自动赚钱"，而是它把交易决策拆成多个角色：

```text
基本面分析师
情绪分析师
新闻分析师
技术分析师
多空研究员
交易员
风控团队
组合经理
```

可以借鉴的地方：

* 多角色分工；
* 多空辩论；
* 风控独立于交易员；
* 组合经理拥有最后审批权；
* 新闻、情绪、技术、基本面分开处理。

不建议照搬的地方：

* 不要让 LLM 直接输出实盘交易；
* 不要因为多个 agent 讨论后结论一致，就认为结论可靠；
* 不要把"语言上的解释充分"当成"统计上有效"；
* 不要让情绪分析直接决定买卖。

适合你的改造方式：

```text
Hermes = 信息分析师 + 研究秘书 + 任务路由
Codex = 工程师 + 审计员
规则系统 = 风控团队
用户 = 组合经理
```

---

### 7.2 Qlib

Qlib 更像完整的 AI 量化研究平台。它适合做：

* 数据处理；
* 模型训练；
* 回测；
* alpha 研究；
* 风险建模；
* 组合优化；
* 订单执行研究。

可以借鉴：

* 完整量化流水线；
* 数据、模型、回测分层；
* 研究和生产之间的接口；
* 自动化研究流程。

不建议一开始完全采用 Qlib 的重型工作流。你现在更适合先做一个可控的小系统，等你理解数据和回测后，再考虑是否迁移或借用 Qlib 的模块。

---

### 7.3 OpenBB

OpenBB 更适合做数据和研究入口。它的价值在于把很多金融数据源统一起来，并提供给 Python、REST API、AI agent、研究面板使用。

可以借鉴：

* 数据统一入口；
* AI agent 友好的数据访问；
* 研究 dashboard；
* 多数据源整合。

它适合放在 Hermes 的数据工具层，而不是直接当交易系统。

---

### 7.4 AKShare / Tushare

如果你要做 A 股或港股相关研究，AKShare 和 Tushare 值得考虑。

适合：

* A 股行情；
* 港股数据；
* 指数；
* 基金；
* 期货；
* 宏观数据；
* 财务数据；
* 公告数据。

注意：

* 免费数据质量要反复校验；
* 字段定义要统一；
* 复权、停牌、退市、财报披露日期要重点检查；
* 不能默认所有数据都适合实盘级使用。

---

### 7.5 vectorbt / backtrader / Zipline / Lean / vn.py

这些项目解决的是不同层次的问题：

```text
vectorbt:
  适合快速、大规模、向量化回测。
  适合研究阶段快速试因子。

backtrader:
  适合事件驱动回测。
  适合更接近真实交易流程的策略模拟。

Zipline / zipline-reloaded:
  适合事件驱动量化研究。
  适合学习经典量化回测结构。

Lean:
  适合完整算法交易系统。
  覆盖研究、回测、优化、实盘部署。

vn.py:
  更偏国内量化交易系统框架。
  适合后续考虑真实交易接口、期货、股票等更完整场景。
```

建议路线：

第一阶段不要纠结"哪个框架最完美"。
先用你最能理解、最容易被 Codex 审计的方式搭建最小系统。

推荐顺序：

```text
先自建最小日频回测 + 模拟账户
再考虑 vectorbt 做快速因子实验
再考虑 backtrader / Lean / vn.py 做更真实交易模拟
```

---

### 7.6 FinRL

FinRL 是强化学习方向的金融交易框架。它有研究价值，但不适合作为你的第一个量化系统核心。

原因：

* 强化学习容易过拟合；
* 结果不容易解释；
* 对环境建模要求很高；
* 对交易成本和市场冲击非常敏感；
* 初学阶段很难判断问题出在数据、环境、奖励函数还是模型。

建议：

> FinRL 可以作为以后学习和实验的对象，不作为第一阶段主线。

## 9. 你需要补充哪些知识

你不需要一开始成为金融专业人士，但必须补以下几类知识。

### 9.1 市场基础

必须知道：

* 股票是什么；
* ETF 是什么；
* 指数是什么；
* 买入价、卖出价、收盘价；
* 市价单、限价单；
* 成交量；
* 流动性；
* 停牌；
* 退市；
* 分红；
* 拆股；
* 复权；
* 交易成本；
* 滑点。

这是最基础的。不懂这些，回测很容易是假的。

---

### 9.2 回测常识

必须知道：

* 未来函数；
* 幸存者偏差；
* 样本内/样本外；
* 过拟合；
* 多重检验；
* 交易成本；
* 滑点；
* 换手率；
* 最大回撤；
* 夏普比率；
* 基准比较。

这是你最应该优先补的。量化新手最容易被漂亮回测骗。

---

### 9.3 因子投资基础

需要知道：

* 什么是因子；
* 什么是 alpha；
* 什么是 beta；
* 什么是 IC；
* 什么是 IR；
* 什么是分组回测；
* 什么是因子衰减；
* 什么是因子中性化；
* 什么是行业中性；
* 什么是市值中性。

你不需要一开始推导复杂公式，但要知道这些概念在回测里怎么用。

---

### 9.4 组合和风控

需要知道：

* 仓位；
* 等权；
* 市值加权；
* 风险平价；
* 波动率目标；
* 最大单股权重；
* 行业暴露；
* 止损；
* 回撤控制；
* 熔断；
* 现金管理。

你的第一阶段重点不是"找到神奇因子"，而是"不因为系统错误亏得不明不白"。

---

### 9.5 数据工程

你本身是计算机背景，这部分会比较容易，但要注意金融数据的特殊性：

* 日期对齐；
* 交易日历；
* 缺失值；
* 复权；
* 时区；
* 财报披露滞后；
* 标的池历史变化；
* 数据版本管理；
* 数据供应商差异。

这部分 Codex 可以帮你很多，但你必须能判断它有没有把金融时间逻辑写错。

## 10. AI 能帮你做什么，不能帮你做什么

### 10.1 AI 能做

Hermes 能做：

* 收集资料；
* 总结新闻；
* 管理笔记；
* 整理策略想法；
* 生成实验卡片；
* 做日报；
* 做周报；
* 维护任务队列；
* 调用 Codex。

Codex 能做：

* 写代码；
* 修 bug；
* 写测试；
* 跑回测；
* 审查未来函数；
* 审查数据泄漏；
* 生成实验报告；
* 重构仓库；
* 建立模拟交易系统。

---

### 10.2 AI 不能做

AI 不能：

* 保证盈利；
* 准确预测市场；
* 替你承担亏损；
* 替代专业投资顾问；
* 判断所有新闻真假；
* 保证数据源正确；
* 保证没有监管和税务问题；
* 在没有约束的情况下安全控制真实资金；
* 自动发现所有隐藏风险。

所以系统设计必须默认：

> LLM 会犯错，数据会出错，回测会骗人，市场会变化。

## 12. 最终建议

当前最合理路线：

1. 用 Hermes 做低成本常驻中枢；
2. 用 Codex MCP Server 做高价值工程子代理；
3. 先搭建最小日频量化系统；
4. 先做 2～3 个月模拟交易；
5. 先验证数据、回测、风控、报告、归档；
6. 不要一开始追求复杂模型；
7. 不要让 LLM 直接下实盘单；
8. 不要把 TradingAgents 这种多代理框架当成赚钱机器；
9. 可以借鉴它的角色分工，但最终交易必须由规则系统和人工确认控制。

一句话总结：

> 你的第一阶段目标不是"让 AI 帮你炒股"，而是"建立一个不会被漂亮回测、混乱数据、无约束代理和错误执行骗倒的量化研究系统"。
